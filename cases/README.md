# cases — реестры случаев

## ladder против ci

- **`ladder/`** — полные ступени верификационной лестницы: боевые
  дискретизации (Q до 1024), замороженные допуски, честная сходимость.
  Гоняются тяжёлыми тестами (маркер `big`; fem-эталоны — маркер `fem`):
  `pytest -m big`.
- **`ci/`** — ЛЁГКИЕ КОПИИ тех же постановок (Q ≤ 256, малые mor_iter):
  каждый файл автоматически становится CI-тестом
  (`tests/test_ci_cases.py` параметризуется содержимым каталога и требует
  `plate-verify` exit 0). Правило: ci-случай обязан проходить БЕЗ
  scikit-fem (эталон fem в ci не используется — ставьте `none` и
  оставляйте fem-сверку ladder-версии).

## Как добавить случай (= тест)

1. `plate-solve --new KIND` → шаблон; заполните постановку.
2. Прогоните и откалибруйте допуск: `plate-verify мой.toml` на боевой
   дискретизации; допуск = 3 × факт (протокол «потолок → факт × 3»),
   диагноз природы ошибки — комментарием в файл.
3. Положите боевую версию в `ladder/`, лёгкую копию — в `ci/`.
   Всё: ci-копия уже тест; для ladder-версии добавьте big-тест
   (пример — `tests/test_analytic_factory.py::test_ladder_new_coverage`).

## Как читать baselines.json

`baselines.json` — ИНФОРМАЦИОННЫЕ регресс-снимки (не путать с допусками
ворот): профили полей, топология зон контакта, калибровочные свипы.
Каждая запись самодокументирована полями «что» и «примечание» (протокол
снимка, платформенная чувствительность, что гейтится, а что только
фиксируется). Снимки сравниваются тестами с кросс-платформенными
допусками; правка снимка — только вместе с породившим её изменением и
записью в CHANGELOG.
