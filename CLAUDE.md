# plate-solver — комплекс программ (изгиб и контакт пластин)

## Контекст
Публичный, самостоятельный программный продукт: изгиб и односторонний
контакт упругих пластин произвольного очертания по уточнённой теории типа
Кармана–Тимошенко–Нагди (КТН); геометрия — R-функциями В. Л. Рвачёва,
контакт — методом обобщённой реакции (МОР). Триада «Модель — Алгоритм —
Программа» (по Самарскому). Держим репозиторий чистым, воспроизводимым
и пригодным для использования посторонним.

## Команды
- Установка: `pip install -e ".[dev]"` (+ `.[dev,fem]` для МКЭ-верификации)
- Тесты: `pytest -m "not big and not fem"` (быстрые); полный — `pytest`
- Стиль: `ruff check .`
- Пример: `python examples/circular_plate.py`
- Эталонный прогон: `python scripts/run_reference.py` (из корня)
- Воспроизводимость: `python scripts/reproduce_all.py` (все case → сводка;
  `--with-ladder` тяжёлые, `--artifacts` поля/рисунки; вывод в results/reproduce)

## Лестница теорий и философия верификации (v0.5.0)
- Ключ `[model] theory`: `classic` (Кирхгоф) | `karman` (геом. нелинейность,
  `membrane.py`) | `ktn_linear` (ЛИНЕЙНЫЕ поправки сдвига/обжатия
  постобработкой) | `ktn_full` (ПОЛНАЯ нелинейная КТН, `ktn_full.py` — слой
  поверх Кармана: члены `−h_*²Δq_n` и `(I−h_ψ²Δ)L`). Устаревшее `ktn` —
  депрекация-алиас на `ktn_linear` (удаление v1.0.0, `docs/MIGRATION.md`).
  НЕ называть линейную поправку именем полной теории; НЕ делать `ktn` алиасом
  на `ktn_full`.
- Лицевые величины — ПЕРВЫЙ КЛАСС (`faces.py`, `Result.thickness_params`), не
  постобработка контакта; канонические имена `h_psi_sq/h_star_sq/h_c_sq` (§3.2).
- Верификация теорий без литературных эталонов (КТН) — РЕДУКЦИОННАЯ лестница
  R1–R5 (`tests/test_ktn_full.py`) + продолжение 1D↔2D; вырождение в подтеории
  (Карман/Кирхгоф) сильнее одного эталона. Кармановские эталоны (Hencky/Way/
  Levy, `benchmarks.py`) — для `karman`; регресс `classic`/`karman`/`ktn_linear`
  не сдвигать.
- Дорожная карта: v0.5.0 (полная КТН) → v0.6.0 (единая параметрическая модель
  теорий + нелинейный контакт МОР поверх КТН + многосвязные области) → v1.0.0
  (консолидация API, упаковка). Единая модель (v0.6.0): все теории —
  ПРЕСЕТЫ параметров одного решателя (`theory.py`, `ktn_solver.py`), редукции
  точны ПО ПОСТРОЕНИЮ; нелинейный контакт — `contact_nl.py` (лицевое условие
  Синьорини, схемы nested|merged). Задел v0.7: маршрутизация контакта через
  case-файлы (`[model] theory` + препятствие), мягкий шарнир для КТН.

## Правила для агента
- IMPORTANT: НЕ ослаблять тест верификации 1D↔2D, чтобы он «прошёл».
  Это сертификат корректности всего метода; при расхождении — искать
  ошибку в методе/краевых условиях, а не в допуске.
- YOU MUST: каждый новый численный метод сопровождать тестом и кратким
  мат. обоснованием (порядок аппроксимации, предполагаемая оценка погрешности).
- Не тащить сюда разовые расчёты, графики «на посмотреть» и тяжёлые данные —
  их место в соседнем репозитории `lab`. Ничего из `data/`, `figures/`,
  `results/`, `*.npz/*.npy/*.csv` не коммитить (см. .gitignore).
- Только свободный инструментарий; зависимости минимальны и зафиксированы.
- Каждый публичный модуль/функция — с docstring; математику пояснять формулой.
- Стиль кода: ruff (line-length 100). Стиль пояснений — академический, по-русски.
- НЕ делать `git add -A` вслепую; не коммитить секреты и личные настройки.
- `private/` НИКОГДА не коммитится и не пушится (домашние материалы автора;
  каталог в .gitignore) — контроль: `git ls-files private/` обязан быть пуст.

## Правила схемы case-файлов и реестров
- Ограда compose-языка: операции union|intersect|difference, примитивы
  circle|rectangle, глубина ≤ 3, ≤ 7 узлов — НЕ расширять без пересмотра
  реестра ворот.
- Новые ключи схемы case-файлов не вводятся без документации (полный
  перечень — docs/CASE_SCHEMA.md; единственный источник соответствий имён —
  GoldenConfig.to_config и Problem.to_config).
- cases/ci/ — только БЫСТРЫЕ случаи (Q ≤ 256, малые mor_iter): каждый
  файл автоматически становится CI-тестом (tests/test_ci_cases.py).
  Тяжёлые ступени — cases/ladder/ (маркер big в их тестах).
- Эталонный отчёт под хеш-воротами: results/reference/reference_v0.4.md
  заморожен (SHA-256, tests/test_reference_hash.py); исторический
  reference_v0.3.md оставлен как есть (вне хеш-ворот). Правка отчёта =
  красный тест, а не повод поправить хеш; обновление — ТОЛЬКО осознанным
  коммитом (перегенерация scripts/run_reference.py) с обоснованием в
  CHANGELOG и новой строкой хеша. golden_config.py (конфиг эталонной
  серии) не трогать.
- Журнал разработки — private/PROGRESS.md: калибровки допусков,
  fallback-решения, отступления от планов.
